# План реализации ADR-0001

> Примечание: этот план относится к superseded-решению из
> [ADR-0001](/home/alko/develop/open-source/websocket1c/docs/decisions/0001-use-sse-transport-for-async-mcp-calls.md).
> Актуальное направление зафиксировано в
> [ADR-0002](/home/alko/develop/open-source/websocket1c/docs/decisions/0002-drop-internal-task-emulation-in-mcp-bridge.md).

## Назначение

Этот документ детализирует внедрение решения из [ADR-0001](/home/alko/develop/open-source/websocket1c/docs/decisions/0001-use-sse-transport-for-async-mcp-calls.md):

- `taskSupport = "required"` требует клиентский task-based вызов;
- `taskSupport = "optional"` при вызове без клиентской задачи использует внутреннюю эмуляцию task lifecycle;
- асинхронный transport для MCP tool calls всегда идёт через `text/event-stream`;
- внутренние задачи имеют специальный префикс идентификатора `internal:`;
- настройка transport-поведения со стороны 1С остаётся простой.

## Объём работ

Входит в план:

- изменение маршрутизации MCP tool calls;
- явное разделение публичных и внутренних задач;
- обновление payload между Rust и 1С;
- корректировка демо-обработчиков;
- обновление документации и тестов.

Не входит в план:

- изменения standalone HTTP/SSE сервера в `src/http/*`;
- переработка `src/http/mcp_handler.rs`;
- изменение внешних endpoint-ов `GET /mcp`, `POST /mcp`, `DELETE /mcp`.

## Целевое состояние

После завершения работ система должна вести себя так:

1. Для `taskSupport = "required"` обычный `tools/call` без task semantics отклоняется.
2. Для `taskSupport = "required"` клиентский task flow через `tasks/*` работает как публичная задача.
3. Для `taskSupport = "optional"` клиентский task flow работает как публичная задача.
4. Для `taskSupport = "optional"` обычный вызов без client task переводится на `text/event-stream` и исполняется как внутренняя задача.
5. Для `taskSupport = "forbidden"` или отсутствующего `taskSupport` сохраняется обычный non-task вызов.
6. Внутренние задачи не попадают в публичные `tasks/list`, `tasks/get`, `tasks/result`, `tasks/cancel`.
7. Внутренние задачи отличаются специальным префиксом идентификатора.
8. Обработчик 1С не обязан вручную выбирать отдельные transport-механизмы для одного асинхронного сценария.

## Порядок реализации

### Этап 1. Зафиксировать модель задач в Rust

Цель:
подготовить кодовую базу к явному разделению публичных и внутренних задач.

Шаги:

1. В `src/mcp/server.rs` ввести явную модель идентификаторов задач.
   - Добавить helper для генерации `task_id`.
   - Вынести специальный префикс внутренних задач `internal:` в константу.
   - Обеспечить единый способ проверки, является задача публичной или внутренней.

2. Уточнить `TaskKind`.
   - Сохранить отдельный тип для публичных задач.
   - Для внутренних задач сохранить отдельный тип, отражающий transport bridge через SSE.

3. Явно отделить код генерации `task_id` для:
   - публичных задач;
   - внутренних задач с префиксом.

Результат этапа:

- идентификаторы внутренних задач формируются централизованно;
- код больше не полагается только на `responseMode` для понимания природы задачи.

### Этап 2. Перестроить маршрутизацию `tools/call`

Цель:
привести серверное поведение в точное соответствие ADR.

Шаги:

1. Обновить правила в `src/mcp/server.rs` для обычного `tools/call`.
   - `required`: plain non-task вызов должен завершаться ошибкой.
   - `optional`: plain вызов должен переходить в асинхронный transport через `text/event-stream` и внутреннюю задачу.
   - `forbidden`: plain вызов должен оставаться обычным синхронным путём.

2. Пересмотреть `request_response_mode` и `ensure_response_mode_allowed`.
   - Явно разделить клиентский `task` и внутреннюю эмуляцию.
   - Исключить двусмысленность, при которой `required` автоматически воспринимается как внутренняя SSE-задача вместо обязательного client task.

3. Обновить ветку, которая сейчас вызывает `call_stream_tool`.
   - Оставить её только для сценариев, совместимых с ADR.
   - Сделать поведение для `optional` и `required` различимым и проверяемым тестами.

Результат этапа:

- логика выбора режима исполнения следует правилам ADR;
- `required` и `optional` перестают смешиваться.

### Этап 3. Упростить bridge с 1С

Цель:
сделать transport-настройку со стороны 1С простой и однообразной.

Шаги:

1. Определить минимальный payload для события `MCP_TOOL_CALL`.
   - 1С должно получать достаточно данных, чтобы отличить:
     - обычный sync-вызов;
     - публичную задачу;
     - внутреннюю эмулированную задачу.

2. В payload добавить явный признак природы задачи.
   - Например, отдельное поле уровня `taskVisibility`, `taskKind` или эквивалент.
   - Не заставлять 1С вычислять это по косвенным признакам.

3. Сохранить простую модель ответа в 1С.
   - Для sync-вызовов: `ОтправитьОтвет`.
   - Для любой task-модели: `ЗавершитьMCPЗадачу`, `УстановитьСтатусMCPЗадачи`, `УведомитьОПрогрессеMCPЗадачи`.

4. Избежать настройки нескольких transport-веток в 1С для одного async-сценария.
   - 1С должно работать с задачей, а не с деталями transport bridge.

Результат этапа:

- со стороны 1С асинхронный сценарий настраивается через task lifecycle;
- transport `text/event-stream` остаётся деталью реализации компоненты.

### Этап 4. Адаптировать демо-обработчик 1С

Цель:
сделать demo отражением новой эталонной семантики.

Шаги:

1. Обновить регистрацию демо-инструментов в `demo/Demo/Forms/Форма/Ext/Form/Module.bsl`.
   - Явно показать примеры для `forbidden`, `optional`, `required`.

2. Обновить обработчик `MCP_TOOL_CALL`.
   - Для `required` отклонять вызов без публичной task semantics.
   - Для `optional` принимать и публичную задачу, и внутреннюю эмуляцию.

3. Обновить сообщения логирования.
   - Отдельно отмечать публичные и внутренние задачи.
   - Логировать специальные префиксы внутренних `taskId`.

4. Проверить совместимость демо-кнопок и вспомогательных клиентских процедур.
   - Если текущие helper-функции в демо жёстко привязаны к старой семантике, адаптировать их.

Результат этапа:

- демо показывает именно ту модель, которую поддерживает реализация;
- документация может ссылаться на демо как на рабочий пример.

### Этап 5. Обновить документацию

Цель:
сделать поведение сервера и документацию непротиворечивыми.

Шаги:

1. Переписать `docs/mcp.md`.
   - Обновить описание `execution.taskSupport`.
   - Описать различие между публичными и внутренними задачами.
   - Указать, что асинхронный transport идёт через `text/event-stream`.
   - Описать специальный префикс внутренних задач.

2. Обновить `docs/architecture/arc42/architecture.md`.
   - Скорректировать архитектурные сценарии и ограничения.
   - Явно зафиксировать модель "SSE transport + public/internal task lifecycle".

3. При необходимости обновить `docs/architecture.md`.
   - Убедиться, что ссылки и краткое описание не конфликтуют с новой моделью.

4. Оставить ADR как источник решения, а этот документ как рабочий план.

Результат этапа:

- документация описывает фактическую модель без противоречий;
- новые участники проекта смогут понять различие между transport и execution semantics.

### Этап 6. Добавить и обновить тесты

Цель:
защитить новую семантику от регрессий.

Шаги:

1. Обновить unit-тесты в `src/mcp/server.rs`.
   - Проверить разрешённые и запрещённые сочетания `taskSupport` и режима вызова.
   - Проверить генерацию внутренних `task_id` с префиксом.

2. Добавить интеграционные тесты для MCP transport.
   - `required` отклоняет plain `tools/call`.
   - `required` работает через публичный task flow.
   - `optional` через plain вызов приводит к SSE transport и внутренней задаче.
   - `optional` через client task приводит к публичной задаче.
   - `forbidden` остаётся обычным вызовом.

3. Проверить скрытие внутренних задач.
   - Внутренние задачи не видны через `tasks/list`, `tasks/get`, `tasks/result`, `tasks/cancel`.

4. Проверить 1С bridge-контракт.
   - Если тестами можно покрыть payload события `MCP_TOOL_CALL`, зафиксировать различие между публичной и внутренней задачей.

Результат этапа:

- новые правила закреплены автотестами;
- регрессии по `taskSupport` и transport проще обнаруживаются.

## Порядок выполнения по коммитам

Рекомендуемая последовательность:

1. Подготовка `src/mcp/server.rs`: идентификаторы, `TaskKind`, helpers.
2. Переключение логики `tools/call` и `enqueue_task`.
3. Корректировка payload и bridge-контракта с 1С.
4. Обновление demo 1С.
5. Обновление документации.
6. Добавление и зелёный прогон тестов.

## Риски

- Смешение старой и новой семантики `required` во время промежуточного состояния.
- Неявная зависимость демо или тестов от старого поведения `stream_sse`.
- Случайное раскрытие внутренних задач через публичные `tasks/*`.
- Переусложнение payload для 1С вместо его упрощения.

## Критерии готовности

План считается выполненным, когда одновременно выполнены все условия:

- кодовая логика соответствует ADR-0001;
- документация и demo описывают ту же модель, что и код;
- внутренние задачи используют специальный префикс;
- async transport идёт через `text/event-stream`;
- со стороны 1С асинхронный сценарий обрабатывается через простой task lifecycle без ручного выбора transport-веток;
- автотесты покрывают `required`, `optional`, `forbidden`, public/internal tasks и SSE transport.
