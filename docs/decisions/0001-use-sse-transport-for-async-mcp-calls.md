# ADR-0001: Использовать `text/event-stream` для асинхронных MCP-вызовов инструментов

- Статус: accepted
- Дата: 2026-03-30
- Приняли решение: maintainer, architecture owner
- Теги: mcp, sse, tasks, transport

## Контекст

MCP-сервер сейчас поддерживает несколько вариантов ответа для `tools/call`: синхронный JSON, `stream_sse` и публичное task-based исполнение через `tasks/*`.

Для долгих инструментов проекту нужна единая серверная модель исполнения, которая позволяет вернуть результат асинхронно и при этом не заставляет всех клиентов использовать только один способ взаимодействия.

Драйверы решения:

- сохранить асинхронное исполнение для долгих обработчиков 1С;
- оставить клиенту выбор там, где это допускает контракт инструмента;
- не дублировать бизнес-логику для вариантов `sync`, `SSE` и `task`;
- сделать транспортный контракт предсказуемым для MCP-клиентов со Streamable HTTP;
- сохранить простую модель bridge между Rust и 1С: с точки зрения компоненты долгий вызов должен выглядеть как задача.

Решение относится только к MCP tool calls. Оно не меняет отдельный HTTP `/sse` endpoint из модуля `http`.

## Решение

Компонента ДОЛЖНА использовать `text/event-stream` как транспорт для асинхронных MCP-ответов инструментов.

`taskSupport` определяет требуемую модель исполнения инструмента:

### `taskSupport = "required"`

- Инструмент требует client-visible task-based вызов.
- Обычный non-task вызов клиента для такого инструмента недопустим.
- Компонента не должна молча подменять это требование чисто внутренней задачей.

### `taskSupport = "optional"`

- Инструмент может вызываться как client-visible task.
- Если клиент не запросил task-based вызов, компонента ДОЛЖНА создать внутреннюю задачу и эмулировать task lifecycle внутри bridge.
- В этом случае внешний ответ всё равно доставляется через `text/event-stream`.
- Идентификаторы таких внутренних задач должны использовать специальный префикс `internal:`, чтобы их можно было однозначно отличать от публичных задач в логах, отладке и внутреннем коде.

### `taskSupport = "forbidden"` или не задан

- Инструмент считается обычным non-task инструментом.
- Task semantics не требуются.
- Вызов обрабатывается как обычный вызов.

Правило транспорта:

- Когда компонента обрабатывает асинхронный MCP-ответ инструмента, HTTP-transport ДОЛЖЕН быть `text/event-stream`.
- Публичные `tasks/*` остаются клиентским контрактом только для инструментов с явно client-visible task semantics.
- Внутренняя эмуляция задачи остаётся деталями реализации bridge для инструментов `taskSupport = "optional"`, вызванных без клиентской задачи.
- Настройка transport-поведения со стороны 1С должна оставаться простой: обработчику 1С не должно требоваться отдельно конфигурировать разные транспортные режимы для одного и того же асинхронного сценария.

## Последствия

Положительные:

- единая внутренняя модель исполнения покрывает статус, прогресс, отмену и завершение;
- инструменты `optional` могут обслуживать и task-aware, и более простых клиентов без дублирования обработчиков;
- клиенты Streamable HTTP получают единый асинхронный транспортный шаблон.

Отрицательные:

- bridge должен явно различать публичные задачи и внутренние эмулированные задачи;
- документация и тесты должны описывать различие между client-visible task и внутренней эмуляцией;
- текущее поведение для `taskSupport = "required"` может потребовать изменения, если сейчас обычные вызовы автоматически переводятся во внутренние SSE-backed задачи.
- потребуется закрепить и последовательно использовать соглашение об именовании внутренних задач.

## Рассмотренные альтернативы

### Требовать публичный task-based flow для каждого асинхронного инструмента

Отклонено.

Это упростило бы семантику, но заставило бы все async-capable клиенты использовать `tasks/*`, даже когда им достаточно SSE-ответа.

### Разрешить каждому инструменту самостоятельно выбирать любой транспорт

Отклонено.

Это сохраняет гибкость, но размывает ожидания клиентов и усложняет bridge между Rust и 1С.

### Сохранить текущее поведение, где `required` может автоматически переключаться во внутреннее SSE-backed task execution

Отклонено.

Такой подход размывает границу между "клиент обязан использовать task semantics" и "компонента может эмулировать задачу внутри", а значит ослабляет смысл `taskSupport = "required"`.

## Не-цели

- переработка standalone HTTP/SSE сервера в `src/http/*`;
- удаление публичных `tasks/*` endpoints из MCP-сервера;
- изменение бизнесовых имён событий 1С без необходимости реализации;
- описание клиентского UX за пределами transport и execution semantics.

## План реализации

Затрагиваемые пути:

- `src/mcp/server.rs`
- `src/mcp/addin.rs`
- `docs/mcp.md`
- `docs/architecture.md`
- `docs/architecture/arc42/architecture.md`
- `demo/Demo/Forms/Форма/Ext/Form/Module.bsl`

Обязательные изменения:

1. Обновить маршрутизацию MCP response mode в `src/mcp/server.rs`.
   - Сохранить `taskSupport = "forbidden"` на обычном non-task пути.
   - Гарантировать, что `taskSupport = "required"` отклоняет обычный non-task `tools/call`.
   - Для `taskSupport = "optional"` при отсутствии публичного task-вызова и необходимости асинхронного завершения переводить транспортный ответ в `text/event-stream` и создавать внутреннюю task entry.

2. Разделить публичный task lifecycle и внутренний эмулированный task lifecycle.
   - Публичные задачи должны оставаться доступными через `tasks/list`, `tasks/get`, `tasks/result` и `tasks/cancel`.
   - Внутренние эмулированные задачи должны оставаться приватными для transport bridge и не попадать в публичные списки задач.
   - Внутренние задачи должны получать специальный префикс идентификатора `internal:`. Префикс должен использоваться единообразно в runtime state, логировании, отладке и проверках.

3. Нормализовать транспортное поведение.
   - Любой асинхронный MCP-ответ, отправляемый компонентой, должен использовать `text/event-stream`.
   - Прогресс и завершение внутренних эмулированных задач должны попадать в SSE-ответ, связанный с исходным вызовом.
   - Со стороны 1С настройка transport-поведения должна оставаться минимальной: для асинхронного инструмента обработчику не нужно вручную выбирать между несколькими внутренними transport-механизмами.

4. Синхронизировать semantics bridge с 1С.
   - Payload, отправляемый в 1С, должен позволять различать публичное task execution и внутреннюю эмуляцию задачи, если это требуется обработчику.
   - Демонстрационные обработчики в `demo/Demo/Forms/Форма/Ext/Form/Module.bsl` должны отражать новую семантику для `required`, `optional` и `forbidden`.

5. Обновить документацию.
   - Переписать `docs/mcp.md`, чтобы он описывал новое значение `taskSupport`.
   - Обновить архитектурную документацию так, чтобы асинхронное поведение MCP описывалось как "SSE transport плюс публичный или внутренний task lifecycle".

6. Обновить тесты.
   - Добавить или скорректировать тесты для сценариев:
     - `required` отклоняет обычные non-task вызовы;
     - `required` работает через публичный task flow;
     - `optional` работает как публичная задача;
     - `optional` без клиентской задачи использует внутреннюю эмуляцию и SSE transport;
     - `forbidden` остаётся на обычном non-task пути;
     - внутренние задачи не публикуются через `tasks/*`.

Ограничения реализации:

- сохранить совместимость со Streamable HTTP;
- не менять поведение несвязанных частей HTTP-модуля;
- держать различие между внутренними и публичными задачами явным в коде, а не только через `responseMode`.

## Проверка

- [ ] Инструмент с `taskSupport = "required"` отклоняет обычный non-task `tools/call`.
- [ ] Инструмент с `taskSupport = "required"` успешно работает через client-visible task flow.
- [ ] Инструмент с `taskSupport = "optional"` может вызываться через client-visible task flow.
- [ ] Инструмент с `taskSupport = "optional"` при вызове без клиентской задачи обрабатывается через внутреннюю эмуляцию task lifecycle.
- [ ] Асинхронный ответ для внутренней эмуляции задачи доставляется как `text/event-stream`.
- [ ] Внутренние эмулированные задачи не попадают в публичные `tasks/list` и `tasks/get`.
- [ ] Внутренние эмулированные задачи используют специальный префикс идентификатора.
- [ ] Инструмент с `taskSupport = "forbidden"` или без `taskSupport` продолжает работать как обычный вызов.
- [ ] `docs/mcp.md` и архитектурная документация описывают ту же семантику, что и реализация.
